# 우진산업 ERP 구축 컨설팅 기록

> 회사: 우진산업 (제지 케미컬)
> 컨설팅: app-mission-architect 에이전트
> 시작일: 2026-05-03

## 배경

- **회사**: 우진산업 (제지 케미컬 회사)
- **현 상황**: 회사에 ERP가 전혀 없음
- **목표**: 직접 ERP 구축
- **제약**: 사용자(빌더)는 ERP 개발 경험이 거의 없음
- **본인이 생각한 모듈 분리안**: 구매 / 생산 / 관리 / 출고 / 회계 (5단계)

## Round 1 — 5단계 분리안 1차 피드백 + 시작 질문

### 에이전트의 1차 피드백

**좋은 점**
- 제조업 ERP의 표준 흐름(구매 → 생산 → 출고 → 회계)을 잘 반영
- 제지 케미컬처럼 "원료 입고 → 배합/생산 → 재고 → 출고 → 정산"으로 흐르는 업종에 자연스러운 분류

**우려되는 점**
- ERP 5개 모듈 동시 구축은 보통 회사가 팀을 꾸려서 1~2년 걸리는 규모. 1인 빌더가 동시 설계 시 90% 이상 중간에 멈추거나 어느 모듈도 안 쓰이는 상태로 끝남.
- "관리"라는 단어가 모호함 — 재고 관리? 거래처 관리? 인사? 의미에 따라 시스템이 완전히 달라짐.
- 더 중요한 건 **"5개 모듈로 나누는 게 맞냐"보다 "지금 회사에서 가장 아픈 곳 하나를 먼저 풀자"**라는 접근이 성공 확률이 훨씬 높음. 한 모듈이 매일 쓰이기 시작하면 자연스럽게 옆 모듈로 데이터가 흘러갈 필요가 생김 — 그때 확장하는 게 정석.

### 에이전트의 1차 질문

**Q1.** 지금 우진산업에서 **하루하루 가장 답답하거나, 시간이 많이 새고 있다고 느끼는 일**이 뭔가요? 구체적인 장면 하나만 떠올려서 말씀해 주세요.
> 예: "어제 거래처에서 재고 얼마 남았냐고 전화 왔는데, 창고 담당자한테 물어보고 엑셀 뒤지고 30분 걸렸다" / "월말마다 매입/매출 정리하느라 이틀이 사라진다" 같은 식

**Q2.** 현재는 이런 일들을 **어떻게** 처리하고 계세요? (엑셀? 수기 장부? 카톡? 거래처별로 다른 양식?) 그리고 회사 규모가 대략 어느 정도인가요 (직원 수, 거래처 수 정도면 충분합니다).

### 사용자 답변

**A1.** 회사가 오래되었지만 중소기업이라 ERP 구축하려면 비용이 너무 크게 들어서 내가 만드려고 함. **아직 전부 다 수기 결제**.

**A2.** 회사 규모
- 직원 약 50명
  - 생산직: 25명
  - 관리영업직: 19명
  - 연구소: 6명
- 연매출: 약 400억
- 생산규모: 연 3만톤급
- 공장: 1공장, 2공장 (2개)

## Round 2 — "수기" 신호 해석 + 통증 좁히기

### 에이전트의 정리

지금까지 들은 내용:
- 50명 규모, 연매출 400억, 공장 2개 운영 중인 중소 제조업체
- **모든 업무가 수기 결재** (시중 ERP는 비용 부담으로 직접 구축 결정)
- 생산직 25명 / 관리영업직 19명 / 연구소 6명 — 이미 작지 않은 조직

> "전부 수기다"는 ERP 컨설팅에서 가장 무거운 신호 중 하나.
> 좋은 소식: 어디를 디지털화해도 효과가 즉시 보임.
> 나쁜 소식: 그렇기 때문에 욕심내서 한 번에 다 하려다 1인 개발자가 무너지는 경우가 정말 많음.
>
> v1은 **딱 한 군데, 가장 피가 많이 흐르는 상처 한 곳**에만 거즈를 대는 식으로 좁혀야 함. 나머지는 v2, v3에서 붙여 나가면 됨.

### 에이전트의 질문

**Q3. 가장 답답한 "한 장면"을 묘사해주세요.**
"모든 업무가 수기"라고 하셨는데, 그 안에서도 분명 유독 자주 터지고, 유독 시간이 많이 빨리고, 유독 실수가 잦은 일 하나가 있을 겁니다. 예시:
- "매일 오후에 생산일보 종이 25장을 받아서 엑셀에 옮겨 적는데 2시간이 사라진다"
- "거래처에서 발주서 오면 재고 확인하러 1공장 2공장 둘 다 전화 돌려야 한다"
- "월말에 출하 내역 모아서 세금계산서 끊을 때 누락이 자주 나서 다시 확인 전화가 폭주한다"
- "결재 서류가 공장장 책상에 며칠씩 쌓여서 진행이 멈춘다"

본인 또는 회사 동료가 실제로 부딪히는 구체적 장면 1~2개. **추상적인 "재고관리"보다 "어제 오후 3시에 무슨 일이 있었는가"가 훨씬 좋은 답.**

**Q4. 본인의 위치와 가용 시간**
- 회사에서 본인의 **역할/직급**? (대표, 관리부장, 생산팀장, 영업담당, IT담당 등)
- ERP 개발에 **일주일에 실질적으로 쓸 수 있는 시간**? (주 5시간인지 20시간인지에 따라 v1 범위가 완전히 달라짐)

### 사용자 답변

**A4. 본인의 위치와 가용 시간**
- 본인은 **3세 대표이사**
- 개발 가용 시간: **주 15시간 정도**
- 개발 기간: 오래 걸려도 됨 — "어차피 평생 내 회사"
- 본인 예상: 부분부분 개발해서 잘 붙이면 **4~6개월**에 초기 버전 가능할 듯

**A3. 가장 답답한 한 장면**
- 회계 정보를 **수기 엑셀로 1명이 다 정리**
- 그 사람이 빠지면 **회사가 잠시 마비**됨 (단일 장애점)
- 다행히 본인이 회계 백그라운드가 있어서 금세 파악은 가능하지만, "뭔가 벽이 있는 느낌"
- ERP로 만들고 싶은 것:
  1. 생산성 관리
  2. 매입 매출
  3. 비용 관리
  4. 급여 관리
  5. 회계 분석 / 경영 진단

> **중요한 시그널**: 사용자의 실제 통증은 "공장 라인 전체 디지털화"가 아니라 "**CEO 시점에서 회계/경영 데이터 가시성 + bus factor 해소**"이다. 처음 제시한 5모듈(구매/생산/관리/출고/회계)과 실제 원하는 것(매입매출/비용/급여/회계분석/경영진단)이 **대부분 회계·경영 영역**임을 주목.

## Round 3 — 처음 가설의 재구성 + v1 제안

### 에이전트의 핵심 통찰

**"한 가지 짚고 가야 할 게 있어요 (정중하게)"**

처음 말한 5단계는 **구매 / 생산 / 관리 / 출고 / 회계** — 전형적인 ERP 교과서 모듈 분류.

그런데 진짜 하고 싶은 5가지:
1. 생산성 관리
2. **매입 매출**
3. **비용 관리**
4. **급여 관리**
5. **회계 분석 / 경영 진단**

5개 중 4개가 사실상 **회계·경영 가시성 영역**. 생산은 1개뿐.

> 대표님의 진짜 통증은 "공장 라인이 안 보인다"가 아니라,
> **"회사 돈이 1명의 머리와 엑셀 안에 갇혀있고, 나조차 벽 너머로 봐야 한다"**
> 그 1명이 내일 그만두면 회사가 마비된다는 게 가장 큰 리스크.

### 에이전트의 v1 제안 (4~6개월)

> **"매입 / 매출 / 비용을 디지털로 옮기고, 대표가 매월 손익을 직접 볼 수 있게 한다"**

- 매입 전표 입력
- 매출 전표 입력
- 비용 항목 입력
- 월별 손익 대시보드 (매출 / 매입 / 비용 / 마진 한눈에)
- **결과: 그 1명이 빠져도 회사가 안 멈춤 + 대표가 숫자를 직접 봄**

확장 로드맵:
- v2: 급여
- v3: 생산 데이터
- v4: 경영 KPI 대시보드

→ 평생 운영할 회사라면 이 페이스가 정석. 4~6개월에 v1 완성 → 매년 1단계씩 확장하면 3~4년 안에 사내 ERP 완성.

### 에이전트의 질문

**Q5. 대표님 직감은 어떠세요?**
매입/매출부터 시작하는 게 맞다고 느끼시나요, 아니면 다른 곳이 더 급하게 느껴지시나요?

**Q6. 회계 담당자 1명에 대해 (UX 결정에 결정적)**
- 지금 **엑셀 시트가 몇 개**고, 어떤 게 매일 쓰는 거고 어떤 게 월말에만 쓰는 건가요? (대충이라도)
- 새 시스템 **자료 입력은 누가 하나요?** 대표님 혼자? 아니면 그 담당자도 같이 쓸 건가요?
  - 대표님 혼자 → 화면이 투박해도 OK, 빠르게 가능
  - 담당자도 같이 → "기존 엑셀보다 불편하면 안 쓴다"는 현실 때문에 UX 신경 많이 써야 함

### 사용자 답변

**A5. 대표님 직감**
- 원료 매입이 가장 시작이고 매출이 가장 마지막 파이프라인
- **양쪽 끝(매입/매출)을 먼저 구성하고 가운데를 브릿지**하려는 접근

**A6. 자료 입력**
- 새 시스템 자료 입력은 **결국 대표님 혼자** 해야 함
- 이유: 현재 회계 담당자가 **정년까지 2년 남음** → 그 시기 내로 ERP 구축이 진짜 데드라인

> **결정적 추가 정보**:
> 1. **진짜 데드라인은 4~6개월이 아니라 2년 (회계 담당자 정년)**. 그 안에 매입~매출 라인 + 회계 분석 + 경영 진단까지 갖춰야 담당자 빠져도 회사 운영 가능.
> 2. 사용자는 **파이프라인 양 끝부터 → 가운데로 수렴**하는 architecture instinct를 이미 가지고 있음. 매입(input)과 매출(output)을 먼저 만들어서 그 사이의 생산/재고/비용 흐름을 채워가는 것 — 이건 1인 빌더에게 유리한 접근.
> 3. **단일 사용자 환경** → UX 미니멀 OK, 빠른 입력만 보장되면 됨.

## Round 4 — 양 끝부터 + 정년 데드라인 = 마스터 플랜

### 에이전트의 핵심 분석

**1) "양쪽 끝부터 → 가운데 브릿지" 접근에 대한 평가**

ERP 모듈 의존성 그래프:

```
[매입] → [재고] → [생산] → [원가] → [재고] → [매출]
  ↑                                            ↓
  └─────────── [비용/급여/회계] ──────────────┘
```

**매입과 매출이 leaf node**, 가운데는 두 끝점을 잇는 중간 노드. 양 끝부터 시작하는 접근의 장점:

- **시스템이 "비어도 동작"**: 가운데가 미완성이어도 양 끝만 있으면 "돈이 얼마 들어와서 얼마 나갔다"는 회계 진실이 잡힘. 단일 빌더에 엄청난 안전장치.
- **데이터 모델 재사용**: 매입과 매출은 본질적으로 같은 "거래 트랜잭션" 패턴 (거래처/품목/수량/단가/일자/결제조건). UI·코드 70~80% 재사용.
- **검증 가능**: 양 끝이 먼저 있으면 가운데 만들 때 "들어가는 거 - 나가는 거 = 가운데가 설명해야 하는 것" reconciliation이 됨.

> **대표님 직감 그대로 진행 권장.**

### 2) 진짜 데드라인 = 회계 담당자 정년 2년

**4~6개월은 v1이 아니라 2년 마스터 플랜의 첫 단계.**

```
[v1: 4~6개월]   매입 + 매출 (양 끝)         ← 지금부터
[v2: ~12개월]   비용 + 급여                   ← 자금 흐름 완성
[v3: ~18개월]   생산/재고 데이터 (가운데)     ← 원가 가시성
[v4: ~24개월]   경영 진단 대시보드            ← 정년 D-day 직전
```

> 이건 단순 ERP 만들기가 아니라 **회계 담당자의 도메인 지식을 시스템으로 옮겨담는 knowledge transfer 프로젝트**.
> 대표님이 직접 입력하는 것도 이 관점에서 합리적 — 입력하면서 도메인을 학습.

### 3) 단일 사용자 = v1에서 뺄 수 있는 것 많음

권한 관리, 멀티 유저 충돌, 화려한 UX, 모바일 반응형, 접근성 → **모두 v1에서 제외 가능**.
시간을 **입력 속도 + 데이터 무결성**에만 집중.

### 에이전트의 마지막 두 질문 (MISSION.md 작성 직전)

**Q7. v1 Anti-scope 확인** — 다음 중 v1에서 빼도 되는 것에 동의하시나요?
- 모바일 앱 (PC 웹/데스크톱만)
- 자동 세금계산서 발행 / 국세청 연동
- 전자결재 / 승인 워크플로우
- 거래처 포털 (외부에서 우진산업 시스템 접속)
- 실시간 재고 추적 (생산 데이터는 v3에서)
- 다중 사용자 / 권한 관리

이 중 "이건 v1에 꼭 있어야 한다" 있으면 알려주세요. 없으면 모두 anti-scope.

**Q8. 매입/매출 입력 단위**
- (a) **전표 단위** — 세금계산서 1장 = 1건 (회계 친화적)
- (b) **거래 단위** — 한 번의 주문/납품 = 1건 (실무 친화적)
- (c) **품목 단위** — 거래 안의 품목 하나하나를 row (분석 친화적, 입력 부담 큼)
- (d) **일자별 합계** — 거래처별 일일 합계만 (입력 빠름, 분석 약함)

힌트: 지금 회계 담당자분이 엑셀에 어떤 단위로 적고 있는지가 그대로 답이 될 가능성 높음.

### 사용자 답변

> Round 4의 Q7/Q8은 명시적 답변 없이, 사용자가 Round 5에서 방향 자체를 좁히는 새 입력을 던짐. 따라서 Q7/Q8은 Round 5 안에서 다시 다뤄야 함.

## Round 5 — 양 끝 동시 → 매입(구매) 한쪽으로 좁힘 + 국내/해외 + 직원 입력 등장

### 사용자의 새 입력

> "구매쪽 국내 해외 구매쪽 DB를 구축하고 우리 직원이 구매내역 입력하고 정리해서 볼 수 있는 ERP의 부분을 만들고 싶어"

### 에이전트의 분석 — 이 한 문장에 담긴 4가지 변화

**1) 양 끝 동시 → 매입(구매) 한쪽으로 더 좁아짐**
- Round 4까지의 합의는 "매입+매출 양 끝부터". 이번엔 매입만.
- 해석 갈림길:
  - (해석 A) "더 작게 시작하자" — v1 안에서 매입을 먼저, 매출은 v1.5/v2로 — **1인 빌더 관점에서 매우 좋은 신호**.
  - (해석 B) 그냥 "매입쪽이 머릿속에서 먼저 떠올랐다" — 여전히 매입+매출 둘 다 v1.
  - 어느 쪽이냐에 따라 4~6개월 v1 범위가 절반으로 줄거나 그대로 유지.

**2) "국내 / 해외" 명시적 구분**
- 이건 데이터 모델에 가장 큰 영향을 주는 문장. 국내 매입과 해외 매입은 같은 "구매"가 아님:
  - **통화**: 국내 KRW 단일, 해외 USD/EUR/CNY/JPY 등 + 환율(매입 시점/결제 시점/장부 환율)
  - **세금**: 국내 부가세 10% / 해외 관세·통관 부가세 (수입신고필증 기준)
  - **결제 조건**: 국내 어음·계좌이체 / 해외 T/T·L/C·D/A·D/P
  - **인보이스 형식**: 국내 세금계산서 / 해외 Commercial Invoice + Packing List + B/L
  - **리드타임**: 국내 며칠 / 해외 수 주 (선적·통관 단계 별도 추적 필요할 수 있음)
- v1에서 둘 다 다루면 데이터 모델이 1.5~2배 복잡해짐. 둘 중 하나만이라도 v1, 나머지는 v1.5로 미루는 게 정석.

**3) "우리 직원이 입력" — 단일 사용자 가정이 깨짐**
- Round 3 A6에서 명시적으로 합의했던 것: "**결국 대표님 혼자**" 입력. 그래서 UX 미니멀 OK라는 대전제가 성립.
- 이번엔 "우리 직원이 입력". 만약 실제로 직원이 v1부터 쓴다면:
  - UX 미니멀 OK 전제 깨짐 — "기존 엑셀보다 불편하면 안 쓴다" 법칙 적용
  - 입력 검증/오류 메시지/실수 복구 UI 필요
  - 다중 사용자/권한 관리는 여전히 anti-scope일 수 있지만, "누가 입력했는지" 기록(audit log)은 v1에 필요
  - 교육·온보딩 비용 발생
- 가능성:
  - (a) 진짜 v1부터 직원 사용 — 위 모두 고려 필요
  - (b) v1은 대표 혼자, v2부터 직원 — 그럼 단일 사용자 전제 유지
  - (c) 입력은 직원, 정리/조회는 대표 — 입력 단순성 + 조회 분석은 별개로 설계
- 셋 중 어느 쪽이냐에 따라 v1 화면 수가 2~3개 늘 수 있음.

**4) "정리해서 볼 수 있는" — 입력 + 조회/리스트뷰가 v1 스코프**
- 이건 자연스러움. 입력만 있고 조회 없으면 ERP가 아니라 입력기. 다만 "정리"의 깊이가 어디까지인지가 v1 스코프를 결정:
  - 단순 리스트 (거래처/날짜/금액 필터·정렬)
  - 거래처별/품목별 합계
  - 월별 추세
  - 환율 적용 후 KRW 환산 합계
  - 미결제·미통관·미입고 추적

### Round 4 Q7/Q8을 이 라운드 안에서 다시 다루는 방식

- **Q7 (Anti-scope)**: 매입 한쪽으로 좁아진 만큼 anti-scope도 더 강해짐. 대신 새 anti-scope 후보 추가 필요 — "v1에서 다중 통화 환산 자동화", "v1에서 수입신고필증 OCR/자동연동", "v1에서 결제 일정 알림".
- **Q8 (입력 단위)**: 매입에 한정하면 후보가 자연스럽게 좁아짐:
  - (a) 세금계산서/Commercial Invoice 1장 = 1건
  - (b) 발주 1건 = 1건 (PO 단위)
  - (c) 품목 단위
  - (d) 입고 단위 (실제로 들어온 시점)
  - 회계 친화는 (a), 실무 친화는 (b)나 (d). "정리해서 본다"의 핵심이 회계 정리면 (a), 발주 추적이면 (b), 재고 흐름이면 (d).

### 에이전트의 질문 (3개로 추림)

**Q9. 직원 입력 시나리오 명확화 (가장 중요)**
지금 "우리 직원이 입력"이라고 하셨는데, 다음 중 어느 쪽에 가장 가까운가요?
- (a) **v1부터 직원이 직접 입력하고 본인은 조회만** — UX·검증·교육 모두 필요
- (b) **v1은 대표님 혼자, v2부터 직원** — Round 3 합의 그대로 유지
- (c) **입력은 직원, 정리/분석 화면은 대표** — 화면이 두 종류로 갈라짐
- (d) **현재 회계 담당자(정년 2년 남으신 분)가 v1 입력** — 가장 도메인 지식 많은 사람이 입력하는 동안 대표가 시스템 학습

→ 이 답에 따라 v1 화면 수, 입력 검증 강도, 권한 처리 여부가 결정됩니다.

**Q10. 국내 / 해외 비중과 우선순위**
대략적인 감으로:
- 우진산업 원료 매입 중 **국내 / 해외 비중**이 어느 정도인가요? (예: 국내 70 / 해외 30)
- **해외 거래처 수**는 대략 몇 곳이고, 어느 나라/통화가 주요한가요? (중국 위안? 유럽 유로? 미국 달러?)
- v1에서 둘 다 가면 데이터 모델이 1.5배 복잡해집니다. 만약 **하나만 먼저**라면 어느 쪽이 더 급하시겠어요?
  - 국내 먼저 (단순, 빠른 v1) → 해외는 v1.5
  - 해외 먼저 (환율·통관까지 한번에) → 국내는 그 위에 얹기
  - 둘 다 동시 (복잡도 감수)

**Q11. 현재 구매 업무의 한 장면**
지금 우진산업에서 원료를 하나 매입할 때, 발주부터 입고·정산까지 **실제로 어떻게** 흘러가는지 한 장면만 묘사해 주세요. 예시 형식:
- "구매팀 ○○ 차장이 거래처에 카톡/전화로 주문 → 거래처 발주확인서 팩스/이메일 → 입고되면 1공장 자재담당이 검수하고 종이 입고증에 사인 → 그 종이가 회계 담당자한테 가서 엑셀에 옮김 → 세금계산서는 한 달 뒤"
- 실제로 **어디서 정보가 끊기고**, **누가 무엇을 들고 있는지**, **어떤 매체(엑셀/카톡/종이/이메일/팩스)** 를 거치는지가 중요합니다.

→ 이 답이 v1 입력 단위(전표/발주/품목/입고)와 데이터 모델의 절반 이상을 결정합니다.
