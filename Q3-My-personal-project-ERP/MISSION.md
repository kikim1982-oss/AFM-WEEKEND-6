# 우진산업 ERP — MISSION

> 작성일: 2026-05-08
> 기반: Round 1~6 컨설팅 기록 (`consulting-log.md`)
> 작성자: app-mission-architect 에이전트

---

## 1. 프로젝트 개요

### 회사 배경
- **회사**: 우진산업 (제지 케미컬)
- **규모**: 직원 약 50명 (생산직 25 / 관리영업 19 / 연구소 6), 연매출 약 400억, 연 3만 톤급, 공장 2개
- **현 상황**: 모든 업무가 수기 결재. 시중 ERP는 비용 부담으로 직접 구축 결정.
- **빌더**: 3세 대표이사 (회계 백그라운드 보유), 주 15시간 가용.

### 진짜 데드라인 — 2년
회계 정보를 수기 엑셀로 1명이 다 정리하고 있고, 그 담당자가 **정년까지 2년 남음**.
즉 이 프로젝트는 단순 ERP 만들기가 아니라 **회계 담당자의 도메인 지식을 시스템으로 옮겨담는 knowledge transfer 프로젝트**.

### 전체 마스터 플랜 (2년 로드맵)

```
[v1: 4~6개월]   매입 (국내 + 해외 원재료)              ← 지금부터
[v1.5/v2]       매출 — 양 끝 완성
[v2~v3]         비용 + 급여 — 자금 흐름 완성
[v3]            생산 / 재고 데이터 (가운데)            ← 원가 가시성
[v4]            경영 진단 대시보드                      ← 정년 D-day 직전
```

대표가 가진 architecture instinct — **양 끝(매입·매출)부터 → 가운데(생산·재고·원가)로 수렴** — 을 그대로 따르되, Round 5에서 사용자 결정에 따라 **v1은 매입 한쪽만** 으로 더 좁힘.

## 2. v1 스코프

### 2.1 v1에 포함되는 것

**대상 모듈**: 매입(구매) — **원재료에 한정**
**거래 종류**: 국내 매입 + 해외 매입 (양쪽 동시)

**핵심 기능 (5가지)**

1. **매입 내역 입력 (국내 양식)**
   - 거래처, 품목, 수량, 단가(KRW), 일자, 결제조건, 세금계산서 번호 등
2. **매입 내역 입력 (해외 양식)**
   - 거래처(해외), 품목, 수량, 단가(외화), 통화, 환율, 환율 적용 시점, Commercial Invoice 번호, B/L, 관세·통관비용, 결제조건(T/T 등) 등
3. **권한 관리 (단순)**
   - 대표가 직원에게 기능별 권한 부여 (입력 권한 / 조회 권한)
   - v1은 **입력자 / 관리자 2단계**만 (세분화 RBAC는 v2 이상)
4. **데이터 조회 / 리스트 뷰**
   - 거래처별 / 품목별 / 일자별 / 통화별 필터·정렬
   - 환율 적용 후 KRW 환산 합계
5. **대표용 분석 / 보고 화면**
   - 1주일 매입 내역 한 화면 조회
   - 거래처별·품목별 합계, 월별 추세
   - **+ Audit log**: 누가 언제 무엇을 입력/수정했는지 기록 (multi-user 진입에 따른 필수 보조 기능)

### 2.2 v1에서 명시적으로 제외 (Anti-scope)

| 카테고리 | 제외 항목 | 이유 |
|---|---|---|
| 모듈 | 매출 전표 | 매입 검증 후 v1.5~v2 |
| 매입 | 부자재 / 소모품 | 원재료만 |
| 회계 연동 | 세금계산서 자동 발행 / 국세청 연동 | 외부 연동 복잡도 |
| 워크플로우 | 전자결재 / 승인 흐름 | 1인 빌더 부담 |
| 채널 | 모바일 앱 | PC 웹/데스크톱만 |
| 채널 | 거래처 포털 | 외부 노출 보안 부담 |
| 데이터 | 실시간 재고 추적 | 생산은 v3 |
| 권한 | 세분화 RBAC | v1은 2단계 (입력자/관리자) |
| 통화 | 다중 통화 환산 자동화 | 수동 환율 입력 |
| 통관 | 수입신고필증 OCR / 자동 연동 | 수동 입력 |
| 알림 | 결제 일정 알림 | v1.5 이후 |

## 3. 사용자 역할

| 역할 | 누가 | 무엇을 |
|---|---|---|
| **입력 담당자** | 직원 (대표가 권한 부여) | 국내·해외 매입 내역 입력, 자기 입력 건 조회 |
| **관리자 / 대표** | 대표이사 본인 | 권한 부여·회수, 전체 조회, 분석/보고 화면, audit log 확인 |

> **중요한 변화**: Round 3까지의 합의는 "v1은 대표 혼자 입력". Round 5에서 사용자가 "직원이 입력, 대표는 조회·분석"으로 명시적으로 뒤집음. 이로 인해 multi-user + 권한 관리가 v1 스코프에 진입.

## 4. 데이터 모델 핵심 항목

### 4.1 공통 (국내 / 해외)
- 거래처 ID, 품목 ID, 수량, 단가, 일자, 입력자 ID, 입력시각, 메모, 첨부 인보이스 파일

### 4.2 국내 매입 고유 항목
- 통화: KRW 고정
- 결제 조건: 현금 / 어음 / 계좌이체 / 외상
- 인보이스: **세금계산서**
- 부가세: 10% (분리 표기)
- 리드타임: 일 단위 (며칠)

### 4.3 해외 매입 고유 항목 (국내와 결정적으로 다른 6가지)

| 항목 | 국내 | 해외 |
|---|---|---|
| **통화** | KRW 단일 | USD / EUR / CNY / JPY 등 |
| **환율 적용 시점** | N/A | 계약일 / 선적일 / 결제일 / 입고일 — **어느 시점 환율을 기록할지 정책 결정 필요** (Open Question) |
| **인보이스 종류** | 세금계산서 1장 | Commercial Invoice + Packing List + B/L |
| **결제 조건** | 현금 / 어음 / 외상 | T/T (전신환) / L/C (신용장) / D/A / D/P |
| **관세·부가세·통관비용** | 부가세만 | 관세 + 통관 부가세 + 운송·하역·창고료 — 별도 라인 vs 매입가 합산 정책 결정 필요 |
| **리드타임** | 며칠 | 몇 주 ~ 몇 달 (선적·통관 단계 별도 추적 필요할 수 있음) |

> v1은 환율·관세를 **수동 입력**으로 처리. 자동 환율 API 연동, 수입신고필증 OCR은 anti-scope.

## 5. 성공 기준 (Measurable)

| 지표 | 목표값 | 시점 |
|---|---|---|
| 국내 매입 1건 입력 시간 | **90초 이내** | v1 출시 시점 |
| 해외 매입 1건 입력 시간 | **3분 이내** | v1 출시 시점 |
| 1주일 매입 내역 조회 | **대표가 한 화면에서** 거래처별·품목별·통화별 합계 확인 가능 | v1 출시 시점 |
| 매입 데이터 디지털 보유율 | **100%** (수기 엑셀 잔존 제로) | 회계 담당자 정년(D-2년) 시점 |
| 직원 1명이 보고 없이 입력 가능 | 교육 30분 이내로 사용 가능 | v1 출시 후 1개월 |

> v1 출시 직후 첫 1주일 매입 내역을 대표가 화면에서 직접 보고 "엑셀 안 봐도 되네"라고 느끼는 순간이 v1 성공의 실질적 신호.

## 6. 일정 및 리스크

### 일정
- **가용 시간**: 주 15시간
- **v1 목표 기간**: 4~6개월 (Round 3 합의 유지)
- **마스터 플랜 데드라인**: 2년 (회계 담당자 정년)

### 리스크 (솔직히 기록)

**1) 권한 관리 + multi-user의 v1 진입에 따른 일정 리스크: +20~30%**
- Round 3까지의 합의는 "단일 사용자". v1 화면 단순·검증 최소·권한 없음 전제로 4~6개월 잡았음.
- Round 5에서 "직원이 입력 + 권한 관리"가 들어오면서:
  - 인증 / 세션 관리
  - 권한 체크 미들웨어
  - audit log
  - 입력 검증 / 오류 메시지 / 실수 복구 UI ("기존 엑셀보다 불편하면 안 쓴다" 법칙)
  - 직원 교육·온보딩
  → 실제 4~6개월이 5~8개월로 늘어날 가능성 있음. **이걸 알고 시작하는 것**이 중요.
- **권장**: v1은 권한을 "입력자 / 관리자 2단계"로만 한정. 부서별·기능별 세분화 RBAC는 v2.

**2) 국내 + 해외 동시 진행에 따른 데이터 모델 복잡도: 1.5~2배**
- 환율, 통관, 다중 통화, 인보이스 형식이 모두 달라 모델·UI·검증 로직이 두 벌.
- 둘 중 하나만 v1으로 가는 옵션도 있었으나, 사용자가 동시 진행을 명시적으로 선택.
- **권장**: 공통 트랜잭션 모델을 먼저 설계하고, 국내·해외는 그 위의 **차이만 분리** (상속 / variant 패턴).

**3) Round 4 Q8 (입력 단위) 미해결**
- 전표 / 거래 / 품목 / 일자별 중 어느 단위로 입력할지 미결정.
- 이건 데이터 모델 절반을 결정하는 사안 → v1 설계 착수 전에 반드시 결정 필요.

**4) Round 5 Q11 (현재 매입 업무 흐름) 미답변**
- 발주 → 입고 → 정산이 실제로 어떻게 흐르는지, 누가 무엇을 들고 있는지 미파악.
- v1 입력 화면을 실제 업무에 맞게 설계하려면 이 정보가 필수 → v1 본격 개발 전에 1~2일 안에 라운드 진행 권장.

## 7. Open Questions (v1 착수 직후 결정 필요)

1. **환율 적용 시점 정책** — 계약일 / 선적일 / 결제일 / 입고일 중 어느 시점 환율을 기록할지. 회계 처리상으로는 결제일 환율이 표준이지만 실무상 입고일도 의미 있음.
2. **거래처 마스터 데이터** — v1에 거래처 마스터를 넣을지. 넣으면 마이그레이션 부담, 안 넣으면 매번 자유 입력으로 데이터 품질 저하.
3. **품목 코드 체계** — 기존 회사 내부 품목 코드가 있는지, 있으면 그대로 쓸지 새로 만들지.
4. **입력 단위** (Round 4 Q8 미해결) — 전표 / 거래 / 발주 / 품목 / 입고 중 무엇을 1건의 단위로 할지. 회계 친화는 전표, 실무 친화는 발주·입고.
5. **현재 매입 업무 실제 흐름** (Round 5 Q11 미답변) — 발주부터 정산까지 어떤 매체(엑셀/카톡/종이/이메일/팩스)를 거쳐 누가 무엇을 들고 있는지.
6. **관세·통관비용 처리 정책** — 별도 라인 vs 매입가 합산. 회계 처리상 매입원가에 포함하는 게 표준이지만 분석 시 분리 가능해야 함.
7. **첨부 파일 처리** — 세금계산서 / Commercial Invoice / B/L 스캔본을 시스템에 보관할지. 보관하면 저장소 비용·검색 기능 필요.
